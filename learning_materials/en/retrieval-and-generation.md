# Retrieval and Generation — From Search to Answer

Once the index is built, the retrieval plane finds relevant chunks and the
generation plane turns them into an answer. Three modules do this work:
`retrieval.py` (cosine search), `prompting.py` (context assembly), and
`generation.py` (LLM call).

---

## Retrieval: Cosine Similarity Search (`retrieval.py`)

### Two functions, one for machines, one for humans

```python
def retrieve(query_text, index, embedder, top_k=5) -> list[RetrievalResult]:
    # Embeds query_text, then delegates to retrieve_by_vector

def retrieve_by_vector(query_vec, index, top_k=5) -> list[RetrievalResult]:
    # Pure ranking — takes a pre-computed vector, returns ranked chunks
```

The split exists for a reason. `retrieve` is the user-facing function — you
pass it natural language and it handles embedding internally. `retrieve_by_vector`
is the test-facing function — you pass it a known vector (e.g., the embedding
of chunk 3 from your fixture corpus) and verify it ranks chunk 3 first.

This separation means you can test retrieval ranking without testing the
embedder. If the ranking is wrong, you know the bug is in the retrieval math,
not in the embedding model.

### How cosine similarity works in this code

The core of `retrieve_by_vector`:

```python
query_vec = np.asarray(query_vec, dtype=np.float32)
q_norm = float(np.linalg.norm(query_vec))
if q_norm == 0.0:
    return []                              # zero query → no results

query_unit = query_vec / q_norm            # normalize query to unit length

emb = index.embeddings.astype(np.float32)  # (N, dim) — all chunk vectors
norms = np.linalg.norm(emb, axis=1, keepdims=True)  # (N, 1)
safe_norms = np.where(norms == 0.0, 1.0, norms)     # avoid division by zero
emb_unit = emb / safe_norms                # normalize all chunk vectors

scores = emb_unit @ query_unit             # (N,) — cosine similarity via dot product
scores[norms[:, 0] == 0.0] = 0.0           # zero-norm chunks get score 0

top_indices = np.argsort(scores)[::-1][:actual_k]   # highest scores first
```

Let's unpack this step by step:

### Step 1: Why normalize at all?

Cosine similarity between two vectors is defined as:

```
cos(a, b) = (a · b) / (||a|| × ||b||)
```

If both vectors are already unit length (||a|| = ||b|| = 1), the denominator
is 1 and cosine similarity reduces to a dot product: `cos(a, b) = a · b`.

The embedders (both fake and real) already produce unit vectors. But the
retrieval code re-normalizes anyway — it's a defense-in-depth measure. If a
future embedder accidentally produces non-unit vectors, retrieval still works
correctly rather than silently producing wrong scores.

### Step 2: The matrix-vector multiplication

`emb_unit @ query_unit` is the key operation. `emb_unit` has shape `(N, dim)`
and `query_unit` has shape `(dim,)`. NumPy broadcasts the dot product across
all N rows, producing a 1-D array of N scores — one cosine similarity per
chunk. This is a single call that does N separate dot products in optimized C,
which is much faster than a Python loop.

### Step 3: Zero-vector handling

The code handles two zero-vector cases:

**Zero query vector.** If the user submits an empty query (or an embedder
produces all zeros for some reason), `q_norm == 0.0` triggers an early return
of `[]`. There's no meaningful way to rank chunks against a zero vector.

**Zero-norm index rows.** If an embedding row is all zeros (shouldn't happen
with real embedders, but the code doesn't trust that), `safe_norms` replaces
the zero with 1.0 to avoid division by zero. After the dot product, those rows
are explicitly set to `score = 0.0`, so they rank last among any ties.

### Step 4: Ranking

`np.argsort(scores)` returns indices sorted by increasing score. `[::-1]`
reverses the array so the highest score comes first. `[:actual_k]` takes the
top k results. The results are then wrapped in `RetrievalResult` with
1-indexed ranks.

### What a good score looks like

With real embeddings, cosine similarities for relevant chunks typically land
between 0.3 and 0.7. A score of 0.9+ is unusually strong; a score around 0.0
means the chunk and query are semantically unrelated. You can inspect these
directly with `rag retrieve`:

```
Rank 1  score=0.6234  chunk_id=a1b2c3d4e5f67890
  Title : API Authentication Guide
  Path  : docs/auth.md
  To authenticate with the watsonx API, you need an API key...
```

## Prompt Assembly: Turning Chunks Into Context (`prompting.py`)

### The prompt template

`prompting.py` contains a visible module-level string — not hidden in
a config file, not generated by a framework. The entire template is one
Python string you can read in 30 seconds:

```python
PROMPT_TEMPLATE = """\
You are a retrieval-augmented assistant. Answer the question using only the
provided context.

If the context is insufficient, say that the provided context does not contain
enough information to answer. Do not use outside knowledge.

Cite every factual claim with the source marker for the context block that
supports it.

Question:
{question}

Context:
{context_blocks}

Answer:"""
```

Four instructions, each chosen for a reason:

| Instruction | What it prevents |
|---|---|
| "Answer using only the provided context" | Model hallucinating from its training data |
| "Say when context is insufficient" | Model fabricating an answer when it shouldn't |
| "Do not use outside knowledge" | Model ignoring retrieved chunks entirely |
| "Cite every factual claim" | Model producing unsupported statements |

### Context block format

Each retrieved chunk becomes one block in the prompt:

```
[Source: a1b2c3d4e5f67890]
Title: API Authentication Guide
Path: docs/auth.md

To authenticate with the watsonx API, you need an API key...
```

The `[Source: <chunk_id>]` marker is the contract between the prompt and the
answer. The model is instructed to include these markers in its response, and
the CLI extracts them with a regex to build the citation list in `AskTrace`.

The same `AskTrace` can also be written to JSON with `rag ask ... --trace-out
PATH`. That JSON contains the full prompt, answer, citations, retrieved chunk
summaries, and stage latencies.

### Why the template is a module-level string

Keeping the prompt template visible in the source code makes it easy to audit,
modify, and experiment with. You don't need to search through config files or
unwrap nested function calls. You just open `prompting.py` and read. This is
intentional: later phases will experiment with different prompt strategies, and
a visible template makes it clear what changed.

## Generation: Calling the LLM (`generation.py`)

### The Generator interface

```python
class Generator(ABC):
    def generate(self, prompt: str) -> str:
        ...
```

One method, one string in, one string out. The pipeline doesn't know or care
whether the answer came from a real API, a local model, or a fake.

### FakeGenerator — deterministic answer for tests

```python
class FakeGenerator(Generator):
    _SOURCE_RE = re.compile(r"\[Source: ([^\]]+)\]")

    def generate(self, prompt: str) -> str:
        markers = self._SOURCE_RE.findall(prompt)
        if not markers:
            return "Based on the provided context, "
                   "the context does not contain enough information to answer."
        cited = " ".join(f"[Source: {m}]" for m in markers)
        return f"Based on the provided context: {cited}"
```

This is clever in a specific way: it scans the prompt for `[Source: ...]`
markers and echoes them back in the answer. This lets tests verify that:

- The prompt correctly included source markers for retrieved chunks.
- The citation extraction regex (`_CITATION_RE`) correctly finds them in the
  answer.
- The full `ask` pipeline — from retrieval through citation — works end to end
  without any network.

When there are no source markers (empty retrieval results), the fake returns an
"insufficient context" message, which tests the abstention path.

### OpenAIGenerator — the real backend

```python
class OpenAIGenerator(Generator):
    DEFAULT_MODEL = "gpt-4o-mini"

    def __init__(self, model=None, api_key=None, base_url=None):
        self._client = OpenAI(api_key=..., base_url=...)

    def generate(self, prompt: str) -> str:
        response = self._client.chat.completions.create(
            model=self.model,
            messages=[{"role": "user", "content": prompt}],
        )
        return response.choices[0].message.content
```

The design is minimal — it wraps exactly one OpenAI SDK call with no
middleware, no retry logic, no streaming. For a learning lab, this is the right
level: you can see exactly what goes to the API and what comes back.

**Configuration priority:** Constructor arguments override environment
variables (`OPENAI_API_KEY`, `OPENAI_BASE_URL`), which override the SDK's
defaults. The CLI layer reads from command-line flags and `os.environ` before
constructing the generator.

**Why `gpt-4o-mini`?** It's cheap, fast, and good enough for a learning lab.
The `--model` flag lets you switch to any OpenAI-compatible endpoint — local
Ollama, Azure OpenAI, or a different cloud provider — without changing any
code.

## What This Teaches

The retrieval-to-generation pipeline is three clean stages:

```
embed query → cosine search → pack prompt → call LLM
```

Each stage is a single function. The contracts are simple types:
`np.ndarray` → `list[RetrievalResult]` → `str` → `str`. There's no shared
state, no global config, no hidden coupling.

The most important concept: **retrieval and generation are separate concerns**.
You can debug retrieval with `rag retrieve` — no LLM involved. You can test
prompt assembly with a fake generator — no network involved. You can switch
embedding models or LLM providers without touching the other. This separation
is what makes RAG debuggable, and it's visible in every line of these three
modules.
